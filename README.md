# Notification Service

Notification service является мостом между брокером сообщений и websocket, поставляя для фронтенда ивенты, идущие от внутренних сервисов. Подобного рода архитектура позволяет разрабатывать «живой» event-based фронтенд, реагирующий на сообщения от различных источников — исходя из того, на какие топики происходит подписка.

Помимо ответственности за websocket слой, notification service имеет собственный api (observer api), позволяющий получать список активных пользователей применимо к какой-либо сущности. На данный момент реализована агрегация подписок по project uuid, что позволяет мониторить, какие участники проекта онлайн, а какие — нет. API и возможности агрегации будут расширены для таких сценариев, как чат, user профиль и т. д. — везде, где имеет место отображение онлайн статуса пользователя.

Используемые технологии - Spring AMQP (RabbitMQ), Spring Web, Spring WebSocket.

## Общая схема
Действующая архитектура предполагает, что все действия пользователя — это rest запросы. На стороне адресата некоторые rest запросы могут запустить процессы, способные генерировать сообщения (ивенты) для внешнего мира. Эти сообщения попадают в rabbitmq, откуда читаются notification сервисом, расшифровывающим контекст и производящим рассылку сообщений в нужные топики. Correlation Id, связывающий все ивенты запущенного пользователем процесса, генерируется фронтендом, наряду с render id – это позволяет разделять не только процессы, но и одну страницу (render) от другой, например в ситуации, когда пользователь открыл одну страницу с двух вкладок.

## Особенности обработки событий из брокера. Расширение стратегии маршрутизации с помощью notification strategy и alarm strategy
Автоматическая маршрутизация событий возможна благодаря тому, что notification сервис уммет расшифровывать контекст, протоколированный для каждой категории ивентов. Каждый exchange в брокере, к которому происходит bind, является источником конкретной категории ивентов. Тип ивентов для notification сервиса не важен.

Расшифровывая контекст, сервис реализует дефолтную стратегию рассылки ивента. Пример — для категорий ProjectEventFromUser и ProjectEventFromSystem происходит рассылка в топик /projects/{uuid}.

Однако не все ивенты можно разослать исключительно в дефолтный топик. Существуют ситуации, когда нам нужно расширить список адресатов для конкретного ивента при конкретном процессе. Для таких случаев в контексте предусмотрено наличие NotificationStrategy и Alarm Strategy (очищаются при передаче на фронтенд), позволяющие гибко настраивать маршрутизацию, при этом сохраняя чистоту notification сервиса, отдавая ему исключительно роль исполнителя.
```
@AllArgsConstructor
@NoArgsConstructor
@Data
public abstract class EventContext {

 // correlation id процесса - каждый процесс в системе должен иметь свой correlation id
 private UUID correlationId;

 private Instant timestamp;

 @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
 private AlarmStrategy alarmStrategy;

 @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
 private NotificationStrategy notificationStrategy;


}
```

Данные стратегии формируются на стороне процесса.

NotificationStrategy содержит два списка с uuid – private channel и public channel. Читая эту стратегию, сервис дублирует ивент в приватный и публичный каналы указанных uuid соответственно.

Alarm Strategy содержит список uuid + тип команды, которую нужно сделать с соединениями, принадлежащими этим uuid. Данный фукнционал позволит, например, обрубать соединения конкретным пользователям в сценариях бана/кика и т.д.


Пример работы со стратегиями опишем ниже.

## Топики
Опишем существующие на данный момент точки подписки websocket (топики):

- /users/activity/private/{uuid} – персональный (приватный) канал пользователя. На него может подписаться только сам пользователь, и никто другой. На данный момент используется для частных уведомлений от некоторых процессов. Пример — удаляется приватный проект. Категория ивентов, связанных с процессом удаления проекта — ProjectEventFromUser. У данного ивента дефолтной стратегией маршрутизации является рассылка в топик /projects/{uuid}, то есть ивент получают текущие подписчики комнаты. Однако о том, что приватный проект удален, должны узнать не только те, кто сидит внутри комнаты, но и тот, кто, к примеру, будучи участником проекта, просматривает страницу со списком всех проектов, в которых он участвует. Чтобы это реализовать, на стороне сервиса — генератора события мы расширяем контекст ProjectEventFromUser, добавляя туда notification strategy с заполненным списком приватных uuid — при расшифровке контекста notification service пошлет событие в приватные топики.
По этой же логике, если бы проект был публичным, notification strategy пришло бы с заполненным списком публичных uuid – тогда об удалении узнали бы все те, кто, например, просто просматривает все проекты автора проекта.

- /users/activity/public/{uuid} – публичный канал пользователя. Сюда присылаются события, о которых могут знать все те, кто просматривает страницу с публичной информацией пользователя. См. Пример выше.

- /project/{uuid} – в данный топик приходят все события внутри комнаты (проекта). Их получают только те, кто находится внутри комнаты.

## Фиксация и индексация подписок
Каждая подписка фиксируется внутри сервиса, что позволяет:
- провести авторизацию — пользователь А не может подписаться на приватный топик пользователя Б;
- структурировать все текущие подписки, давая возможность извлекать их по тому или иному параметру (индексирование)

Websocket соединение состоит из двух этапов:
- Рукопожатие — обычный http запрос, перехватываемый и обогащаемый фильтром в api gateway, что позволяет notification сервису знать, кто является инициатором соединения
- Непосредственная подписка на топики, происходящая в рамках установленной сессии.

Перед тем, как перейти с схеме обработки websocket соединения, рассмотрим интерфейс и устройство Observer сервиса.
> Данный сервис на данный момент является прототипом, в будущем он должен быть переработан в угоду безопасности и масштабируемости — вероятно, мигрирует в redis

Как видно из схемы, мы разделяем такие сущности, как «Сессия» и «Подписка». В Stomp одна сессия может иметь множество разных подписок, что требует гибкого индексирования по типу подписки.

Все сессии хранятся в хеш-таблице sessionStore в формате Session Envelope, содержащей данные о том, кому принадлежит сессия, а также spring объект сессии, который потенциально можно использовать для ручного управления жизненным циклом соединения.


Рассмотрим алгоритм добавления подписки - на входе мы знаем только session id и адрес топика. На основе этих данных мы должны сформировать Subscription объект, после чего вставить его в соответствующий индекс. Для этого мы парсим адрес, извлекая payload данные. Далее мы проверяем, существует ли в sessionStore объект сессии. Если да — заполняем security параметры объекта subscription.

После этого вставляем подписку в подходящее хранилище, при этом все подписки попадают в userToSubAssociation – это является обязательным индексом, необходимым для оптимизации операции удаления всех подписок, ассоциированных с сессией.

Пример:
Подписка на -/project/uuid. Объект Subscription вставляется как в userToSubAssociation, так и в projectToSubAssociation.

Как уже было упомянуто, удаление (unregister) сессии происходит вместе с удалением всех ассоциированных с этой сессией подписок. Удаляя сессию, мы извлекаем из нее user uuid, после чего обращаемся к userToSubAssociation. Мы извлекаем из него все подписки, ассоциированные с user uuid, после чего, используя payload, очищаем от этих подписок все остальные индексы. Подобный подход значительно ускоряет операцию удаления — нам не нужно проводить сложные операции поиска с перебиранием списков, в каждом из которых нужно найди подписку с session id.

addSubscription и unregisterSecuredSession сопровождаются публикацией websocket событий, таких как PROJECT_SUB И PROJECT_UNSUB – это дает возможность фронтенду быть реактивным, то есть обновлять данные только при получении точечного ивента, не полагаясь на дорогое обновление каждые несколько секунд.
## Схема обработки Websocket соединений
> Нижеописанная схема не является окончательной, так как игнорирует этап авторизации. Для реализации авторизации необходимо использовать иной interceptor.

> В частности, вместо @EventListener нужно использовать ChannelInterceptor, позволяющий отклонять подписки, несоответствующие security context. Также открытым остается вопрос — как валидировать доступ к сущностям, лежащим вне пределов notification service.

> Дальнейшие соображения для модификации — изменения должны быть совместимы с логикой observer service, соответствующем парадигме, где понятие Session отделено от подписки.

Для генерации security context на данный момент мы используем компонент CustomHandShakeInterceptor, реализующий интерфейс HandshakeInterceptor. Этот компонент добавляется в список interceptor’ов при регистрации stomp endpoint’ов в конфигурации WebSocketMessageBrokerConfigurer.

В методе beforeHandshake мы получает доступ к ServerHttpRequest, в котором мы читаем заголовки, вставленные API Gateway при валидации запроса. Кстати, именно на этом этапе было решено перейти к хранению токенов в http-only cookie (вместо local storage), так как модификация handshake запроса на фронтенде оказалась невозможной.

Извлеченные заголовки мы сохраняем в объект SecurityContext, после чего вставляем его в атрибуты сессии (Map <String, Object>), доступные нам наравне с ServerHttpRequest. Так сгенерированный контекст становится видимым для следующего шага цепочки.

Следующим шаг прописывается в вышеупомянутом WebSocketMessageBrokerConfigurer, для которого переопределяется метод configureWebSocketTransport. 

Добавляется кастомный WebSocketHandlerDecorator, где переопределяются методы afterConnectionEstablished(WebSocketSession session) и afterConnectionClosed(WebSocketSession session, CloseStatus close). Эти методы являются точкой связи с observer методом, где в первом случае происходит регистрация сессии с сохранением ссылки на объект session, а во втором — удаление сессии с последующей очисткой ее подписок.

Для добавления подписки реализован обыкновенный @EventListener, слушающий событие SessionSubscribeEvent. Мы извлекаем данные о session id и topic, после чего вызываем addSubscription(String sessionId, String path) в observer сервисе.
